# 称号模块设计方案

## 一、概述

将现有 `common.py` 中 `CoreService._title_rules()` 的被动称号逻辑拆出为独立的 `修仙/称号/` 二级包，与异火、探险等模块平级。称号无属性加成，纯装饰，作为消息头和人物志的近况窗口。

核心变更：
- 保留现有 27 个被动称号的获取规则，仍由各玩法结算时被动刷新称号池。
- 追加 30 个新称号，覆盖等级、宗门、首领、虫洞、对战、二手市场、异火、藏宝图等维度。
- 新增玩家可手动选择佩戴称号的命令入口，打破"只自动佩戴最高优先级"的限制。

## 二、命令入口

| 命令 | 作用 |
|---|---|
| 称号 | 查看当前佩戴称号、已拥有称号数量、入口按钮 |
| 称号列表 | 查看所有称号分类和获取状态概览 |
| 称号详情 称号名 | 查看单个称号的获取途径、描述和佩戴状态 |
| 称号佩戴 称号名 | 手动佩戴已拥有的某个称号 |
| 称号卸下 | 卸下手动佩戴的称号，恢复为自动佩戴模式 |
| 称号图鉴 | 查看全部称号的锁定/解锁状态和获取进度 |

## 三、展现形式

## 四、数据库

### 现有表 `player_titles`

沿用，不改结构：

```sql
CREATE TABLE IF NOT EXISTS player_titles (
    client_id TEXT NOT NULL,
    title TEXT NOT NULL,
    reason TEXT NOT NULL DEFAULT '',
    active INTEGER NOT NULL DEFAULT 0,
    obtained_at TEXT NOT NULL,
    updated_at TEXT NOT NULL,
    PRIMARY KEY (client_id, title)
);
```

### 新增表 `title_prefs`

保存玩家手动佩戴偏好：

```sql
CREATE TABLE IF NOT EXISTS title_prefs (
    client_id TEXT PRIMARY KEY,
    manual_title TEXT NOT NULL DEFAULT '',
    updated_at TEXT NOT NULL
);
```

- `manual_title` 为空时，表示使用自动佩戴模式。
- `manual_title` 不为空时，消息头优先显示该称号。

## 五、称号规则设计

### 5.1 设计原则

1. 称号无属性加成，纯装饰。
2. 称号通过被动数据判定获取，不新增主动领取入口。
3. 现有 27 个称号规则保持不变。
4. 新 30 个称号按游戏现有数据源判定，不依赖尚未实现的系统。
5. 称号分为 10 个类别，每类 3~8 个，由易到难。

## 六、模块结构

```text
修仙/称号/
├── __init__.py      # WS 命令注册
├── service.py       # 称号业务逻辑
└── 说明.md          # 组件文档
```

### `__init__.py` 命令注册

```python
@WsMessageHandler.handler(cmd="称号", priority=100, block=True)
async def ws_title_my(client_id, message): ...

@WsMessageHandler.handler(cmd="称号列表", priority=100, block=True)
async def ws_title_list(client_id, message): ...

@WsMessageHandler.handler(cmd="称号详情", priority=100, block=True)
async def ws_title_detail(client_id, message): ...

@WsMessageHandler.handler(cmd="称号佩戴", priority=100, block=True)
async def ws_title_equip(client_id, message): ...

@WsMessageHandler.handler(cmd="称号卸下", priority=100, block=True)
async def ws_title_unequip(client_id, message): ...

@WsMessageHandler.handler(cmd="称号图鉴", priority=100, block=True)
async def ws_title_codex(client_id, message): ...
```

### `service.py` 核心逻辑

- `my_titles(client_id)` — 查看当前佩戴和已拥有数量。
- `list_all(client_id)` — 按分类展示全部称号和获取状态。
- `detail(client_id, message)` — 单个称号详情。
- `equip(client_id, message)` — 手动佩戴已拥有的称号，写入 `title_prefs`。
- `unequip(client_id)` — 清除手动偏好，恢复自动佩戴。
- `codex(client_id)` — 称号图鉴，已解锁/未解锁状态和获取进度。
- `refresh_titles(client_id, player)` — 被动刷新称号池（从 common.py 迁移）。
- `_title_rules(stats)` — 称号规则（从 common.py 迁移并扩展）。

## 七、与其他组件的联动

| 组件 | 联动方式 |
|---|---|
| reply.py | 读取称号时，先查 `title_prefs.manual_title`，非空则用它；否则查 `player_titles` 的 `active = 1` |
| common.py | `refresh_titles()` 和 `_title_rules()` 迁移到称号模块的 service；common.py 保留薄包装调用 |
| 玩家/修仙信息 | 结算时调用 `refresh_titles()` 刷新称号池 |
| 探险/跑商/对战/首领/虫洞 | 结算时调用 `refresh_titles()` 刷新称号池 |
| 修仙界历史/人物志 | 读取当前称号展示 |
| 清理 | 删除不存在玩家时清理 `title_prefs` |

## 八、迁移方案

1. 新建 `修仙/称号/` 目录和文件。
2. 将 `common.py` 中 `_title_rules()`、`_title_stats_conn()`、`refresh_titles()`、`refresh_titles_conn()`、`active_title()`、`_save_valid_titles_conn()` 迁移到 `称号/service.py`。
3. `common.py` 中保留 `refresh_titles()` 和 `active_title()` 的薄包装，内部调用称号模块 service。
4. 新增 `title_prefs` 表。
5. `reply.py` 中读取称号的逻辑改为先查 `title_prefs`。
6. 各组件中调用 `self.refresh_titles()` 的地方改为调用称号模块 service。

## 十、与现有称号系统的兼容

- 现有 27 个称号的获取条件、优先级和 reason 文案完全不变。
- `player_titles` 表结构不变，`ON CONFLICT` 语义不变。
- 新增 `title_prefs` 表仅存手动偏好，不影响自动佩戴逻辑。
- `reply.py` 读取称号顺序：`title_prefs.manual_title` → `player_titles.active = 1` → `无`。
- 各组件结算时仍调用 `refresh_titles()` 刷新称号池，手动佩戴的称号不会被自动刷新覆盖。
