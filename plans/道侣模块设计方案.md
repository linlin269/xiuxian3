# 道侣模块设计方案

## 一、业务规则总结

### 性别系统
- `players` 表新增 `gender TEXT NOT NULL DEFAULT ''` 字段
- 创建用户时必须选择性别：男 / 女
- 之后可通过纳戒物品"阴阳道果"变性，该物品属于角色成长道具，价格处于商城物品价格中上游
- 性别未设置前（空字符串）不允许使用道侣功能

### 道侣功能开关
- 默认关闭，需发送 `道侣功能 开启` 打开
- 关闭后不收到他人申请，也不能发起申请
- 通过 `道侣功能 关闭` 关闭

### 结道侣规则
- 必须一男一女
- 一人只能同时拥有一个道侣；已有道侣时不能再次申请
- 申请方式类似切磋：A 发起申请 → B 接受/拒绝
- 申请有效期 24 小时
- 不保留历史申请记录，只保留当前待处理申请，处理后直接删除

### 解除道侣
- 任意一方都可单方面发起解除
- 解除后双方恢复为无道侣状态

### 黑名单
- **手动拉黑**：发送指令拉黑指定用户，永久不再收到该用户的结道侣申请，直到自己手动移出黑名单
- **自动拉黑**：同一用户申请未接受（超时或拒绝）满 3 次，永久进入自动拉黑，手动移出黑名单不清零历史累计次数
- 解除道侣不影响黑名单状态

## 二、数据库扩展

### 2.1 players 表新增字段

```sql
ALTER TABLE players ADD COLUMN gender TEXT NOT NULL DEFAULT '';
```

### 2.2 新增表：dao_lu_requests（道侣申请）

```sql
CREATE TABLE IF NOT EXISTS dao_lu_requests (
    request_id INTEGER PRIMARY KEY AUTOINCREMENT,
    from_client_id TEXT NOT NULL,
    to_client_id TEXT NOT NULL,
    status TEXT NOT NULL DEFAULT '等待',
    created_at TEXT NOT NULL,
    expires_at TEXT NOT NULL
);
CREATE INDEX IF NOT EXISTS idx_dao_lu_requests_to ON dao_lu_requests(to_client_id, status);
CREATE INDEX IF NOT EXISTS idx_dao_lu_requests_from ON dao_lu_requests(from_client_id, status);
```

### 2.3 新增表：dao_lu_partners（道侣关系）

```sql
CREATE TABLE IF NOT EXISTS dao_lu_partners (
    client_id TEXT PRIMARY KEY,
    partner_id TEXT NOT NULL,
    paired_at TEXT NOT NULL
);
```

### 2.4 新增表：dao_lu_blacklist（黑名单）

```sql
CREATE TABLE IF NOT EXISTS dao_lu_blacklist (
    owner_id TEXT NOT NULL,
    blocked_id TEXT NOT NULL,
    reason TEXT NOT NULL DEFAULT '手动',
    blocked_at TEXT NOT NULL,
    PRIMARY KEY (owner_id, blocked_id)
);
```

### 2.5 新增表：dao_lu_application_stats（申请统计）

```sql
CREATE TABLE IF NOT EXISTS dao_lu_application_stats (
    from_client_id TEXT NOT NULL,
    to_client_id TEXT NOT NULL,
    total_count INTEGER NOT NULL DEFAULT 0,
    last_applied_at TEXT NOT NULL,
    auto_blocked INTEGER NOT NULL DEFAULT 0,
    PRIMARY KEY (from_client_id, to_client_id)
);
```

| `修仙信息` | 展示中增加性别和道侣信息 |

|---|---|

- 性别

- 道侣功能开关状态

## 六、清理策略

- 过期道侣申请（`expires_at` 已过）在下次操作时懒清理，删除记录
- 道侣申请记录不进入 `game_logs` 清理周期，因为不保留历史
- 黑名单和申请统计永久保留，不做清理

## 七、组件联动

- **创建用户**：玩家组件需扩展性别参数
- **修仙信息/状态**：展示性别和道侣信息
- **纳戒**：阴阳道果消耗入口
- **通知系统**：道侣申请待处理通知
- **回复系统**：道侣组件按钮配置加入 `CONTEXT_BUTTONS_BY_GROUP`

## 十、风险点

1. **性别字段迁移**：已有玩家的 gender 默认为空字符串，需要在修仙信息中提示设置性别
2. **创建用户兼容**：老命令 `创建用户 名称` 不带性别时，需要提示补充性别
3. **并发申请**：同一玩家同时收到多人申请时，需要检查是否有待处理申请
4. **解除道侣时的黑名单**：解除道侣不影响黑名单，但需要确认不影响自动拉黑计数
5. **阴阳道果使用限制**：已有道侣时不能使用，需要在纳戒 service 中增加校验
