### 总体判断

英灵殿模块已经覆盖了“刷新 NPC → 查看列表 → 发起挑战 → 结算奖励 → 查看记录”的主链路，但当前实现存在多处高风险缺陷，其中包含**会直接导致模块不可用的运行时错误**、**奖励公式偏差**、**挑战记录不准确**以及**调度器启动不稳**等问题。建议先修复关键缺陷后再进入联调或对外开放。

## 二、主要问题清单

### 1. P0：`require_player()` 调用方式与基类签名不一致，模块入口会直接报错

**问题描述**

[`CoreService.require_player()`](xiuxian3/xiuxianserver/修仙/common.py:643) 的定义是：

```python
def require_player(self, client_id: str) -> tuple[dict[str, Any] | None, str | None]:
```

但英灵殿实现中在多个入口直接写成了：

- [`overview()`](xiuxian3/xiuxianserver/修仙/英灵殿/service.py:181)
- [`npc_list()`](xiuxian3/xiuxianserver/修仙/英灵殿/service.py:224)
- [`challenge()`](xiuxian3/xiuxianserver/修仙/英灵殿/service.py:276)
- [`records()`](xiuxian3/xiuxianserver/修仙/英灵殿/service.py:452)

都调用为：

```python
self.require_player(conn, client_id)
```

**影响**

这会直接触发 `TypeError`，导致英灵殿相关命令入口不可用。

**结论**

这是当前最严重的问题，属于上线阻断级别。

---

### 2. P0：经验奖励公式使用了累计经验，和设计方案不一致

**设计要求**

设计方案要求胜利后获得“玩家当前升级所需经验的 0.5%”，对应应当按单级需求计算。

**实现现状**

[`challenge()`](xiuxian3/xiuxianserver/修仙/英灵殿/service.py:385) 使用的是：

```python
exp_reward = max(
    1,
    int(player_exp_for_level(int(player["level"]) + 1) * HALL_EXP_RATIO),
)
```

但 [`player_exp_for_level()`](xiuxian3/xiuxianserver/修仙/rules.py:107) 计算的是“达到指定等级的累计经验”，不是“升一级所需经验”。

**影响**

奖励会明显偏大，且等级越高偏差越严重，破坏数值平衡。

**建议**

应改为按 [`exp_need()`](xiuxian3/xiuxianserver/修仙/rules.py:80) 计算当前等级升到下一级所需经验，再乘以 `HALL_EXP_RATIO`。

---

### 3. P0：NPC 仅按名称挑战，缺少稳定唯一标识

**问题描述**

NPC 名称由 [`ChineseName.generate()`](xiuxian3/xiuxianserver/修仙/英灵殿/service.py:128) 随机生成，挑战时却仅按名字查询：

```python
SELECT * FROM hall_of_heroes_npcs WHERE name = ? AND defeated = 0
```

**影响**

- 同批次或跨批次出现重名时，玩家无法可靠指向目标
- 查询只会命中第一条，存在误挑战风险
- `records()` 依赖 `npc_id` 关联，但挑战入口没有提供稳定 NPC 定位方式

**建议**

至少需要在输出列表中附带 `npc_id` 或批次内序号，并允许命令按编号挑战；更稳妥的是让挑战指令支持“名字 + 编号”双重定位。

---

### 4. P1：战败分支没有回写 `hp_left`

**问题描述**

挑战失败时，当前逻辑只执行：

```python
UPDATE players SET mp = 0 WHERE client_id = ?
```

没有同步写回剩余血气。

**影响**

- 若战斗因回合上限耗尽而失败，玩家血气状态会残留为旧值
- 战斗结果与玩家当前状态不一致
- 后续状态判断可能受到影响

**建议**

失败分支应统一回写战后 `hp_left` 和 `mp_left`，至少保持结算结果与玩家状态一致。

---

### 5. P1：挑战记录字段与实际结果不一致

**问题描述**

在胜利记录中：

- `damage_dealt` 写入的是 [`highest_damage`](xiuxian3/xiuxianserver/修仙/英灵殿/service.py:358)
- `recover_item_id` 写成了逗号拼接字符串
- `recover_quantity` 固定写 `1`

**影响**

- 伤害统计失真，无法准确反映战斗结果
- 恢复道具记录无法精确拆分
- 后续如果做排行、补偿、分析或审计，会得到错误数据

**建议**

建议拆分为结构化字段，例如：

- `damage_dealt`：总伤害
- `recover_item_id`：主回血道具 ID
- `recover_quantity`：实际数量
- 另外可增加 JSON 字段保存完整奖励明细

---

### 6. P1：刷新清理逻辑与“保留战果”设计存在偏差，且会造成表膨胀

**设计要求**

设计方案写明：已击败 NPC 应保留到下一批刷新前，用于展示战果，但不重复挑战。

**实现现状**

[`refresh_npcs()`](xiuxian3/xiuxianserver/修仙/英灵殿/service.py:145) 只删除：

```python
DELETE FROM hall_of_heroes_npcs WHERE defeated = 0
```

这意味着已击败 NPC 永远不会被清理。

同时 [`npc_list()`](xiuxian3/xiuxianserver/修仙/英灵殿/service.py:224) 只展示当前批次，旧战果实际上不会在列表中被看到。

而 [`records()`](xiuxian3/xiuxianserver/修仙/英灵殿/service.py:452) 通过 `JOIN hall_of_heroes_npcs` 展示历史记录，如果未来清理 NPC 表，还会影响历史可读性。

**影响**

- `hall_of_heroes_npcs` 会持续增长
- “保留战果供查看”的体验没有真正落地
- 历史记录展示过度依赖 NPC 主表，数据生命周期设计不完整

**建议**

建议明确两层数据：

1. 当前批次 NPC 表，只保留当前批次或最近一批
2. 战斗历史表，保存挑战快照，避免依赖当前 NPC 记录

---

### 7. P2：调度器在模块导入时直接启动，且吞掉异常

**问题描述**

[`__init__.py`](xiuxian3/xiuxianserver/修仙/英灵殿/__init__.py:45) 中使用：

```python
scheduler.start()
```

并且异常被直接 `pass` 掉。

**影响**

- 导入时 event loop 状态不合适，可能导致任务根本未成功启动
- reload 或重复导入时容易出现不可预期行为
- 启动失败被静默吞掉，不利于排错

**建议**

将调度器生命周期纳入应用启动/停止阶段统一管理，不要在模块 import 时直接启动。

---

### 8. P2：`chinesename` 依赖未锁定版本

**问题描述**

[`requirements.txt`](xiuxian3/xiuxianserver/requirements.txt:9) 中只有：

```text
chinesename
```

**影响**

部署可重复性弱，不同环境可能拉到不同版本，导致命名行为变化。

**建议**

锁定版本号，并在部署环境中验证兼容性。

## 三、设计与实现对照结论

### 已基本落实的内容

以下设计已在实现中体现：

- 每小时刷新 10 名 NPC
- NPC 稀有度随机分配
- 基于固定基值 + 首领公式 + 稀有度比例生成数值
- 单人挑战与战斗结算
- 镇渊积分奖励接入 [`grant_npc_points_conn()`](xiuxian3/xiuxianserver/修仙/镇渊诛邪/service.py:614)
- 恢复药品进入纳戒
- 经验奖励与挑战记录
- WS 命令注册

### 仍存在设计缺口或实现偏差的内容

- 挑战对象的唯一标识不足
- 历史 NPC/战果展示规则未闭环
- 奖励经验公式与方案不一致
- 战败状态回写不完整
- 调度器生命周期管理不规范

---

## 四、建议修复优先级

### 优先级 P0

1. 修正 [`require_player()`](xiuxian3/xiuxianserver/修仙/common.py:643) 的调用方式，确保模块入口可用
2. 修正经验奖励公式，改为按 [`exp_need()`](xiuxian3/xiuxianserver/修仙/rules.py:80) 计算
3. 为 NPC 增加稳定唯一标识，避免只靠名字挑战

### 优先级 P1

4. 补全战败分支的 `hp/mp` 回写
5. 重构挑战记录结构，保证伤害和奖励统计准确
6. 调整 NPC 刷新保留策略，避免数据无限增长

### 优先级 P2

7. 将调度器启动迁移到应用生命周期管理
8. 锁定 [`chinesename`](xiuxian3/xiuxianserver/requirements.txt:9) 版本

---

## 五、审查结论

英灵殿模块的整体结构是完整的，但当前版本**不建议直接上线**。其中最关键的阻断问题是 [`require_player()`](xiuxian3/xiuxianserver/修仙/common.py:643) 调用错误和经验奖励公式偏差，这两项会直接影响功能可用性与数值正确性。建议先按本报告的 P0/P1 项修复后，再进行一次回归审查与联调验收。
