# 镇渊与诛邪模块设计方案

## 一、文档目标

本文档用于定义镇渊与诛邪模块的最终设计口径，作为后续进入 `code` 模式实施的依据。

本方案重点解决以下问题：

- 镇渊与诛邪的等级展示与升级关系
- 镇渊提供的基础数值、恢复速度、探险速度、黑市折扣的计算方式
- 诛邪提供的独立乘区与叠加加成方式
- 击杀 NPC 数量的开放接入方式
- 普通怪击杀来源的开放接入方式，确保未来刷怪副本可复用同一口径
- 与人物面板、休息、探险、黑市、战斗属性结算的联动边界
- 数据结构、命令入口与验收标准

本文以 [`plans/镇渊与诛邪模块.docx`](plans/镇渊与诛邪模块.docx) 为原始设定来源，并根据当前系统中黑市、探险、玩家面板与数值结算的实现风格进行收口。

---

## 二、核心设计结论

### 2.1 镇渊与诛邪的显示关系

- 镇渊达到 10 级后，不再同时展示镇渊 10 与诛邪 1。
- 当镇渊进阶到诛邪后，界面只显示当前主阶段名称。
- 展示规则建议如下：
  - 镇渊 0 - 9：显示 `镇渊 X`
  - 镇渊 10：显示 `镇渊 10`
  - 诛邪 1 - 10：显示 `诛邪 X`
- 系统内部仍需保留两个概念：
  - `zhenyuan_stage`：镇渊阶段，0 到 10
  - `zhuxie_stage`：诛邪阶段，0 到 10
- 但在对外展示时，按当前活跃阶段只显示一个名称，不做双标题并列。

### 2.2 镇渊属性的定位

镇渊提供的是**独立追加到人物对应数值上的基础值**，不是百分比乘区，也不替代其他模块。

镇渊提供的四项基础值：

- 血量基础值
- 防御基础值
- 攻击基础值
- 精神基础值

这四项会直接加到玩家人物的最终对应数值上，属于额外固定增益。

镇渊同时提供三类功能性加成：

- 恢复速度：影响玩家执行 `休息` 指令时减少的时间
- 探险速度：影响玩家执行 `探险` 指令后到探险结束减少的时间
- 黑市折扣：影响购买黑市物品时获得的价格折扣

### 2.3 诛邪属性的定位

诛邪提供的是**作用于镇渊基础值的独立乘区**，不是直接作用于人物主属性的通用乘区。

诛邪的四项乘区：

- 血量乘区
- 防御乘区
- 攻击乘区
- 精神乘区

结算时先计算镇渊基础值，再乘以诛邪乘区，最后把结果作为**单独增加值**加到玩家对应属性里。

诛邪的三项功能性加成：

- 恢复速度
- 探险速度
- 黑市折扣

这三项是在镇渊 10 的基础值上继续叠加，不替代镇渊原有效果。

### 2.4 开放接口原则

镇渊与诛邪的积分来源不能写死在单一模块中，必须保留开放接口，便于后续新增来源。

当前需要明确预留两类开放接入口：

- **NPC 击杀数量接口**：用于宗门、任务、活动或未来其他系统上报 NPC 击杀数
- **普通怪击杀数量接口**：普通怪当前指探险中的怪物，未来若开发专门刷怪副本，也应通过同一接口接入

镇渊模块只消费统一的积分接入结果，不直接关心积分来自哪个具体玩法模块。

## 三、最终规则定稿

### 3.1 镇渊成长规则

#### 3.1.1 升级方式

- 镇渊通过累计积分升级。
- 积分来源与现有战斗、虫洞、首领、NPC 结算体系挂钩。
- 镇渊 0 级为初始状态。
- 镇渊 10 级为满级。
- 镇渊 10 后将进入诛邪成长阶段。

#### 3.1.2 积分换算

沿用原文档的统一换算口径，并补充开放接口约束：

- 1 点积分 = 1 次普通怪击杀
- 1 点积分 = 0.02 次虫洞 BOSS 击杀
- 1 点积分 = 0.02 次首领击杀
- 1 点积分 = 0.2 次 NPC 击杀

等价表达：

- 50 积分 = 50 次普通怪击杀 = 1 次虫洞 BOSS = 1 次首领 = 5 次 NPC 击杀

补充说明：

- **普通怪** 当前特指探险中的普通怪物。
- 未来若新增专门刷怪副本，只要通过“普通怪击杀接口”上报，仍按同一口径计入镇渊积分，不需要修改镇渊规则本身。
- NPC 击杀数必须由外部模块通过开放接口上报，镇渊模块不直接扫描各模块内部战斗明细。
- 虫洞 BOSS、首领等高价值来源仍可走统一积分入口，但其来源模块应保留独立标识，便于审计和后续扩展。

### 3.3 诛邪成长规则

#### 3.3.1 升级方式

- 当镇渊达到 10 后，继续升级转入诛邪阶段。
- 诛邪从 1 级开始，不再显示镇渊 10 与诛邪 1 的并列状态。
- 诛邪 10 为当前上限。

#### 3.3.2 结算原则

诛邪只影响镇渊提供的基础值，不影响其它来源的属性。

例如：

- 镇渊 10 提供攻击基础值 200
- 诛邪 1 提供攻击乘区 1.02
- 最终追加攻击 = 200 × 1.02 = 204
- 这个 204 作为单独加值加入玩家攻击力
- 不会进一步作用到装备、称号、其他系统的增益上

血量、防御、精神同理。

## 四、数值结算规则

### 4.1 总体结算顺序

建议采用以下顺序计算人物相关属性：

1. 先计算玩家自身基础属性
2. 再叠加镇渊提供的基础值
3. 再对镇渊基础值应用诛邪乘区
4. 最后合并其他模块对人物最终属性的影响

### 4.2 血量、防御、攻击、精神结算

#### 4.2.1 镇渊基础值

镇渊提供固定值，直接加到玩家对应属性：

- 玩家最终血量 = 玩家原始血量 + 镇渊血量基础值 + 其他来源固定值
- 玩家最终防御 = 玩家原始防御 + 镇渊防御基础值 + 其他来源固定值
- 玩家最终攻击 = 玩家原始攻击 + 镇渊攻击基础值 + 其他来源固定值
- 玩家最终精神 = 玩家原始精神 + 镇渊精神基础值 + 其他来源固定值

#### 4.2.2 诛邪乘区

诛邪只乘镇渊基础值，不乘玩家全部属性。

建议公式：

```text
镇渊追加值 = 镇渊基础值 × 诛邪乘区
最终追加值 = floor 或四舍五入后的镇渊追加值
```

其中四项分别独立计算。

示例：

- 镇渊 10 攻击基础值 = 200
- 诛邪 1 攻击乘区 = 1.02
- 镇渊攻击追加值 = 200 × 1.02 = 204
- 再把 204 作为独立数值加到玩家攻击力

### 4.3 恢复速度结算

恢复速度有两层含义：

- 镇渊的恢复速度：镇渊 10 的基础恢复缩短值
- 诛邪的恢复速度：在镇渊 10 基础上继续叠加

建议结算方式：

- 休息恢复时间基准 = 当前系统定义的标准休息时间
- 镇渊缩短 = 基准时间 × 镇渊恢复速度比例
- 诛邪追加缩短 = 基准时间 × 诛邪恢复速度比例
- 最终休息时间 = 基准时间 - 镇渊缩短 - 诛邪追加缩短

如果实现中采用剩余时间减少模式，也应保证语义一致：

- 镇渊先减少一部分
- 诛邪在镇渊 10 的基础上再减少一部分

### 4.4 探险速度结算

探险速度指玩家执行 `探险` 指令后，到探险结束所减少的时间。

建议规则：

- 镇渊探险速度先作用于基础探险时长
- 诛邪探险速度再叠加缩短
- 只影响探险时长，不影响战斗判定和掉落概率

### 4.5 黑市折扣结算

黑市折扣指玩家购买黑市物品时的折扣。

建议规则：

- 镇渊 10 提供基础折扣
- 诛邪在此基础上继续叠加
- 折扣只影响黑市购买，不影响黑市回收价
- 回收价仍按售价比例计算，不叠加玩家折扣

示例：

- 某商品售价 10000
- 镇渊 10 折扣 5%
- 诛邪 1 追加折扣 1%
- 最终购买价 = 10000 × (1 - 0.05 - 0.01)
- 实际结算时向下取整到源石整数

## 五、展示与交互设计

### 5.1 面板展示原则

建议在玩家信息面板中新增镇渊/诛邪区块，展示内容包括：

- 当前阶段名称
- 当前等级与上限
- 当前积分/下一级所需积分
- 四项基础值或乘区
- 恢复速度
- 探险速度
- 黑市折扣

## 六、数据结构设计

### 6.1 推荐数据库表

建议新增独立成长表，例如：

```sql
CREATE TABLE IF NOT EXISTS player_zhenyuan_zhuxie (
    client_id TEXT PRIMARY KEY,
    zhenyuan_stage INTEGER NOT NULL DEFAULT 0,
    zhuxie_stage INTEGER NOT NULL DEFAULT 0,
    zhenyuan_points INTEGER NOT NULL DEFAULT 0,
    zhenyuan_base_hp INTEGER NOT NULL DEFAULT 0,
    zhenyuan_base_def INTEGER NOT NULL DEFAULT 0,
    zhenyuan_base_atk INTEGER NOT NULL DEFAULT 0,
    zhenyuan_base_spirit INTEGER NOT NULL DEFAULT 0,
    zhenyuan_recovery_bonus REAL NOT NULL DEFAULT 0,
    zhenyuan_explore_bonus REAL NOT NULL DEFAULT 0,
    zhenyuan_black_market_discount REAL NOT NULL DEFAULT 0,
    zhuxie_hp_multiplier REAL NOT NULL DEFAULT 1,
    zhuxie_def_multiplier REAL NOT NULL DEFAULT 1,
    zhuxie_atk_multiplier REAL NOT NULL DEFAULT 1,
    zhuxie_spirit_multiplier REAL NOT NULL DEFAULT 1,
    zhuxie_recovery_bonus REAL NOT NULL DEFAULT 0,
    zhuxie_explore_bonus REAL NOT NULL DEFAULT 0,
    zhuxie_black_market_discount REAL NOT NULL DEFAULT 0,
    updated_at TEXT NOT NULL
);
```

### 6.2 推荐积分来源日志表

为支持开放接口，建议新增来源日志表，记录各模块上报的积分来源。

```sql
CREATE TABLE IF NOT EXISTS player_zhenyuan_zhuxie_point_logs (
    record_id INTEGER PRIMARY KEY AUTOINCREMENT,
    client_id TEXT NOT NULL,
    source_type TEXT NOT NULL,
    source_module TEXT NOT NULL,
    source_key TEXT NOT NULL DEFAULT '',
    source_name TEXT NOT NULL DEFAULT '',
    quantity INTEGER NOT NULL,
    point_ratio REAL NOT NULL,
    point_value INTEGER NOT NULL,
    created_at TEXT NOT NULL,
    extra TEXT NOT NULL DEFAULT ''
);
```

字段说明：

- `source_type`：来源类型，如 `normal_monster`、`npc_kill`、`wormhole_boss`、`boss` 等
- `source_module`：上报来源模块，如 `探险`、`首领`、`异界虫洞`、`宗门` 等
- `source_key`：具体对象稳定标识，例如地点、怪物、NPC 或副本标识
- `quantity`：击杀或结算数量
- `point_ratio`：单次数量对应的积分比例
- `point_value`：最终入账积分
- `extra`：扩展字段，便于未来记录更详细的上下文

### 6.3 开放接口建议

建议在服务层统一提供“积分接入接口”，而不是由镇渊模块直接读取各系统内部数据。

建议接口语义如下：

- `record_normal_monster_kill(...)`：普通怪击杀接入
- `record_npc_kill(...)`：NPC 击杀接入
- `record_boss_kill(...)`：虫洞 BOSS / 首领等高价值目标接入
- `grant_growth_points(...)`：通用积分写入接口

接口原则：

- 普通怪当前由探险模块调用
- 未来刷怪副本只要调用同一接口即可，无需修改镇渊本体
- NPC 击杀来源由对应玩法模块上报，镇渊只关心积分结果，不关心来源细节
- 来源类型必须可审计、可扩展、可区分

### 6.4 字段说明

- `zhenyuan_stage`：镇渊阶段，0 到 10
- `zhuxie_stage`：诛邪阶段，0 到 10
- `zhenyuan_points`：当前累计积分
- `zhenyuan_base_hp / def / atk / spirit`：镇渊对应基础值
- `zhenyuan_recovery_bonus`：镇渊恢复速度
- `zhenyuan_explore_bonus`：镇渊探险速度
- `zhenyuan_black_market_discount`：镇渊黑市折扣
- `zhuxie_hp_multiplier / def / atk / spirit`：诛邪四项乘区
- `zhuxie_recovery_bonus`：诛邪恢复速度追加值
- `zhuxie_explore_bonus`：诛邪探险速度追加值
- `zhuxie_black_market_discount`：诛邪黑市折扣追加值
- `updated_at`：最近更新时间

### 6.5 推荐分层

建议将数据拆成三层处理：

- **定义层**：等级表、积分阈值、数值表
- **状态层**：玩家当前阶段、积分、已激活加成
- **结算层**：在玩家面板、探险、黑市、休息时实时读取并应用

## 七、与现有模块的联动规则

### 7.1 与玩家模块

玩家模块负责：

- 展示镇渊/诛邪面板
- 读取当前阶段与积分
- 汇总最终血量、防御、攻击、精神
- 汇总恢复速度、探险速度、黑市折扣

建议不要把镇渊/诛邪的完整玩法逻辑塞进玩家主逻辑中，而是通过服务接口读取结果。

### 7.2 与探险模块

探险模块需要读取：

- 探险速度加成
- 普通怪积分接入回调
- 是否触发与阶段有关的提示文案

镇渊/诛邪不改变探险的战斗机制，只影响探险时长计算。

### 7.3 与黑市模块

黑市模块需要读取：

- 当前黑市折扣总和
- 是否需要对购买价做折扣

黑市回收价保持独立规则，不受镇渊/诛邪影响。

### 7.4 与休息模块

休息模块需要读取：

- 当前恢复速度总和
- 用于计算休息时长缩减

镇渊与诛邪只影响 `休息` 指令对应的时间，不影响战斗内即时恢复效果的其他规则，除非后续另行约定。

### 7.5 与未来刷怪副本模块

未来若开发专门刷怪副本：

- 副本中击杀的“普通怪”应通过 `record_normal_monster_kill(...)` 接口接入
- 副本不应直接修改镇渊主表
- 副本只需维护自己的掉落、战斗与结算逻辑
- 镇渊模块只消费其上报的统一积分结果

## 八、计算示例

### 8.1 镇渊 10 示例

假设玩家当前拥有：

- 镇渊 10
- 诛邪 0

则其加成为：

- 血量 +1000
- 防御 +150
- 攻击 +200
- 精神 +100
- 恢复速度 +20%
- 探险速度 +10%
- 黑市折扣 -5%

### 8.2 镇渊 10 + 诛邪 1 示例

假设玩家当前拥有：

- 镇渊 10 的攻击基础值 200
- 诛邪 1 的攻击乘区 1.02

则：

- 镇渊攻击追加值 = 200 × 1.02 = 204
- 这 204 直接加到玩家攻击力
- 其他三项同理独立计算
- 恢复速度、探险速度、黑市折扣则在镇渊 10 基础上继续叠加

### 8.3 黑市折扣示例

假设：

- 镇渊折扣 = 5%
- 诛邪折扣 = 1%
- 商品售价 = 10000

则购买价建议按：

- 折扣总和 = 6%
- 实际价格 = 10000 × 0.94 = 9400
- 最终结算向下取整

## 九、实施建议

### 9.1 代码拆分建议

建议新增二级组件：`修仙/镇渊诛邪/`

```text
修仙/镇渊诛邪/
├── __init__.py
├── service.py
└── 说明.md
```

其中：

- `__init__.py`：注册命令入口
- `service.py`：镇渊与诛邪业务逻辑、数值计算、面板输出
- `说明.md`：本模块开发说明

### 9.2 公共能力复用建议

建议复用或扩展以下共用能力：

- 数值裁剪与安全取整
- 百分比格式化
- 时间缩减计算
- 玩家面板汇总拼装
- 积分来源统计
- 开放接口接入层

### 9.3 不建议事项

- 不建议把镇渊和诛邪拆成两个完全独立模块，否则会导致阶段衔接复杂
- 不建议把诛邪乘区设计为全局乘区，否则会污染其它系统
- 不建议把黑市折扣和回收价混为一谈
- 不建议在展示层同时显示镇渊 10 与诛邪 1 两套标题
- 不建议把普通怪和 NPC 击杀直接硬编码在镇渊模块里，必须通过开放接口接入

## 十、验收标准

### 10.1 功能验收

1. 镇渊达到 10 后，继续升级只显示诛邪阶段，不显示双标题。
2. 镇渊的血量、防御、攻击、精神四项基础值会独立加到玩家对应属性。
3. 镇渊的恢复速度只影响 `休息` 指令时长。
4. 镇渊的探险速度只影响 `探险` 指令时长。
5. 镇渊的黑市折扣只影响黑市购买价。
6. 诛邪的四项乘区只作用于镇渊基础值。
7. 诛邪的恢复速度、探险速度、黑市折扣在镇渊 10 基础上叠加。
8. 黑市回收价不受镇渊/诛邪折扣影响。
9. 玩家面板可正确展示当前阶段、数值和折扣总和。
10. NPC 击杀数量可通过开放接口接入，不需要修改镇渊主逻辑。
11. 未来刷怪副本中的普通怪击杀可通过同一开放接口接入。

### 10.2 数值验收

- 镇渊 10 攻击基础值 200，诛邪 1 乘区 1.02 时，最终追加攻击为 204。
- 镇渊 10 恢复速度 20%，诛邪 1 恢复速度 +3%，最终恢复速度为 23%。
- 镇渊 10 探险速度 10%，诛邪 1 探险速度 +2%，最终探险速度为 12%。
- 镇渊 10 黑市折扣 5%，诛邪 1 黑市折扣 +1%，最终黑市折扣为 6%。

### 10.3 边界验收

- 阶段值不得越界到负数或超上限。
- 黑市折扣结算不得产生负购买价。
- 诛邪乘区不得回乘玩家全局主属性。
- 镇渊基础值不得重复叠加。
- 普通怪积分必须保留来源标识，方便未来副本接入和审计。

## 十一、总结

本方案将镇渊与诛邪明确拆分为“基础值层 + 独立乘区层 + 功能速度层 + 开放来源层”的结构：

- 镇渊负责把玩家的基础面板抬高
- 诛邪负责在镇渊基础值上继续放大
- 恢复速度、探险速度、黑市折扣作为功能型成长，按阶段叠加
- 普通怪击杀与 NPC 击杀统一通过开放接口接入，避免未来扩玩法时重构镇渊逻辑
- 展示层只保留当前活跃阶段名称，避免镇渊 10 与诛邪 1 同屏重复

这样设计后，数值口径清晰，和现有黑市、探险、玩家面板系统的联动也更容易落地。
